## Estilização em Next.js 16: CSS Modules, Tailwind v4 e Bootstrap

> **Versões fixadas:** `next@16.2.4`, `tailwindcss@4.2.4`, `@tailwindcss/postcss@4.2.4`, `bootstrap@5.3.8`, `sass@1.89.0`.

O Next.js 16 suporta várias estratégias de CSS de forma nativa: **CSS Global**, **CSS Modules**, **Sass/SCSS**, **CSS-in-JS** (via bibliotecas), **Tailwind CSS v4** e, se você ainda quiser, frameworks CSS tradicionais como **Bootstrap**. Vamos focar nas abordagens mais recomendadas hoje.

```mermaid
flowchart TB
    A[App Router Next.js 16] --> G[Global CSS<br/>app/globals.css]
    A --> M["CSS Modules<br/>*.module.css"]
    A --> S["Sass<br/>*.scss / *.module.scss"]
    A --> T[Tailwind CSS v4]
    A --> CSS[CSS-in-JS<br/>(styled-components, Emotion...)]
    A --> B[Frameworks prontos]
    B --> BS[Bootstrap 5]
    B --> DS[Design Systems<br/>shadcn/ui, Radix, HeadlessUI...]
    classDef recommended fill:#e0f8e0,stroke:#080;
    class T,M,G,S recommended;
```

> **Recomendação geral**: para projetos novos, combine **Global CSS mínimo** + **CSS Modules** para estilos locais, ou use **Tailwind CSS v4** (padrão do `create-next-app`). CSS-in-JS é menos interessante no App Router porque exige configurações de SSR e aumenta o bundle do cliente. Materialize-CSS, popular há alguns anos, **não é recomendado para Next.js 16**: depende de jQuery e manipula o DOM globalmente (quebra com Server Components). Foi removido deste material.

### 1. CSS Global e CSS Modules (built-in)

#### 1.1. Global CSS

Importe apenas **uma vez** no `app/layout.js` (ou nos layouts raiz de route groups):

```css
/* app/globals.css */
:root {
  --color-primary: #2c3e50;
}

body {
  margin: 0;
  font-family: system-ui, sans-serif;
}
```

```jsx
// app/layout.js
import './globals.css';

export default function RootLayout({ children }) {
  return (
    <html lang="pt-BR">
      <body>{children}</body>
    </html>
  );
}
```

#### 1.2. CSS Modules (escopo local automático)

Crie um arquivo `*.module.css` ao lado do componente:

```css
/* app/_components/Button.module.css */
.button {
  background: var(--color-primary);
  color: white;
  padding: 0.5rem 1rem;
  border-radius: 4px;
  border: none;
  cursor: pointer;
}

.button:hover {
  opacity: 0.9;
}
```

```jsx
// app/_components/Button.js
import styles from './Button.module.css';

export default function Button({ children, onClick }) {
  return (
    <button className={styles.button} onClick={onClick}>
      {children}
    </button>
  );
}
```

Vantagens: classes são automaticamente prefixadas (não colidem), excelentes para componentes reutilizáveis, e funcionam tanto em Server quanto Client Components.

#### 1.3. Sass / SCSS

Next.js tem suporte nativo se você instalar:

```bash
npm install --save-dev sass@1.89.0
```

Basta renomear arquivos para `.scss` ou `.module.scss` — o Next reconhece automaticamente.

### 2. Tailwind CSS v4 (abordagem recomendada)

Tailwind v4 mudou bastante: não há mais `tailwind.config.js` obrigatório, a detecção de arquivos é automática e as diretivas `@tailwind base; @tailwind components; @tailwind utilities;` foram substituídas por um simples `@import "tailwindcss";`.

#### 2.1. Usando o `create-next-app` (jeito mais fácil)

Ao criar um projeto com `--tailwind`, tudo é configurado:

```bash
npx create-next-app@16.2.4 my-tw-app \
  --javascript --tailwind --no-eslint --no-src-dir --app --turbopack \
  --import-alias "@/*" --use-npm
cd my-tw-app
npm install
```

O template já inclui:

```json
// package.json (trecho)
"devDependencies": {
  "@tailwindcss/postcss": "^4",
  "tailwindcss": "^4",
  "babel-plugin-react-compiler": "1.0.0"
}
```

```js
// postcss.config.mjs
const config = {
  plugins: {
    "@tailwindcss/postcss": {},
  },
};
export default config;
```

```css
/* app/globals.css */
@import "tailwindcss";

:root {
  --background: #ffffff;
  --foreground: #171717;
}

@theme inline {
  --color-background: var(--background);
  --color-foreground: var(--foreground);
}
```

Pronto! Agora é só usar classes utilitárias:

```jsx
// app/page.js
export default function Home() {
  return (
    <main className="min-h-screen bg-gray-100 flex items-center justify-center">
      <div className="rounded-lg bg-white p-8 shadow-lg">
        <h1 className="mb-4 text-3xl font-semibold text-blue-600">
          Bem-vindo ao meu e-commerce
        </h1>
        <button className="rounded-lg bg-blue-500 px-4 py-2 text-white hover:bg-blue-700">
          Clique aqui
        </button>
      </div>
    </main>
  );
}
```

#### 2.2. Instalando Tailwind v4 manualmente em um projeto existente

Se você criou um projeto **sem** `--tailwind`, instale fixando as versões:

```bash
npm install --save-dev tailwindcss@4.2.4 @tailwindcss/postcss@4.2.4
```

Crie `postcss.config.mjs` na raiz:

```js
// postcss.config.mjs
const config = {
  plugins: {
    '@tailwindcss/postcss': {},
  },
};

export default config;
```

Edite `app/globals.css`:

```css
/* app/globals.css */
@import "tailwindcss";

@theme {
  --color-brand: #2563eb;
}
```

Certifique-se de que `app/layout.js` importa esse arquivo (o default do `create-next-app` já faz isso).

Use a classe custom criada no `@theme`:

```jsx
<h1 className="text-brand">Título com a cor da marca</h1>
```

#### 2.3. Principais diferenças da v3 para a v4

| v3                                                                 | v4                                                              |
| ------------------------------------------------------------------ | --------------------------------------------------------------- |
| `tailwind.config.js` (obrigatório)                                 | Configuração via `@theme` no CSS (config JS opcional)            |
| `@tailwind base; @tailwind components; @tailwind utilities;`       | `@import "tailwindcss";`                                         |
| Plugin PostCSS: `tailwindcss` + `autoprefixer`                     | Plugin único: `@tailwindcss/postcss` (autoprefixer embutido)     |
| `content: [...]` manual                                            | Auto-detecção baseada nos imports                                |
| Modo JIT opcional                                                  | JIT é o único modo (e é instantâneo com Oxide/Turbopack)         |

### 3. Bootstrap 5 (quando usar um framework CSS "clássico")

Bootstrap continua sendo útil para protótipos rápidos. Como depende de JS opcional (dropdowns, modais), precisa de cuidado: os componentes JS dele **não funcionam em Server Components** — por isso isole-os em Client Components.

#### 3.1. Instalar fixando versão

```bash
npm install bootstrap@5.3.8
```

#### 3.2. Importar o CSS global

```jsx
// app/layout.js
import 'bootstrap/dist/css/bootstrap.min.css';
import './globals.css';

export default function RootLayout({ children }) {
  return (
    <html lang="pt-BR">
      <body>{children}</body>
    </html>
  );
}
```

#### 3.3. Usar classes em qualquer componente

```jsx
// app/page.js
export default function Home() {
  return (
    <div className="container py-5">
      <h1 className="text-center mb-4">Bem-vindo ao meu e-commerce</h1>
      <button className="btn btn-primary">Clique aqui</button>
    </div>
  );
}
```

#### 3.4. Componentes JS do Bootstrap (modais, tooltips, dropdowns)

Esses dependem do `bootstrap/dist/js/bootstrap.bundle.min.js`, que mexe com `window/document`. Carregue-os **apenas em Client Components** usando `useEffect`:

```jsx
// app/_components/InteractiveModal.js
'use client';

import { useEffect, useRef } from 'react';

export default function InteractiveModal() {
  const ref = useRef(null);

  useEffect(() => {
    let modalInstance;
    (async () => {
      const bootstrap = await import('bootstrap/dist/js/bootstrap.bundle.min.js');
      modalInstance = new bootstrap.Modal(ref.current);
    })();

    return () => modalInstance?.dispose();
  }, []);

  return (
    <>
      <button
        className="btn btn-secondary"
        onClick={() => {
          // A instância já existe após o useEffect
          // para simplicidade, o toggle seria via API do bootstrap.
        }}
      >
        Abrir modal (exemplo)
      </button>
      <div className="modal fade" ref={ref} tabIndex="-1" aria-hidden="true">
        <div className="modal-dialog">
          <div className="modal-content">
            <div className="modal-body">Conteúdo</div>
          </div>
        </div>
      </div>
    </>
  );
}
```

> Para quem quer Bootstrap "React-friendly" sem essa complicação, considere [`react-bootstrap`](https://react-bootstrap.github.io/) — só lembre que ele precisa ser usado em Client Components.

### 4. CSS-in-JS (styled-components, Emotion) no App Router

É possível, mas exige um _StyledComponentsRegistry_ que capture estilos durante o SSR para que não ocorra FOUC. Consulte o [guia oficial](https://nextjs.org/docs/app/guides/css-in-js). Para a maioria dos projetos novos, **evite** CSS-in-JS em Server Components — prefira CSS Modules ou Tailwind.

---

### 5. Comparativo e recomendação

| Abordagem         | Custo de setup | JS no cliente | Suporte a Server Components | Recomendado para…                             |
| ----------------- | -------------- | ------------- | --------------------------- | --------------------------------------------- |
| Global CSS        | zero           | zero          | sim                         | Temas, resets, variáveis CSS                  |
| CSS Modules       | zero           | zero          | sim                         | Componentes isolados em qualquer projeto      |
| Sass/SCSS Modules | baixo          | zero          | sim                         | Projetos que querem nesting/mixins            |
| Tailwind CSS v4   | baixíssimo (template pronto) | zero | sim              | **Maioria dos novos projetos**                |
| Bootstrap 5 (CSS) | baixo          | zero          | sim                         | Protótipos rápidos, reuso de componentes prontos |
| Bootstrap 5 (JS)  | médio          | médio         | só em Client Components     | Uso pontual de modais/dropdowns               |
| CSS-in-JS         | médio/alto     | médio/alto    | exige registry              | Apenas se for pré-requisito do projeto        |
| Materialize CSS   | (descontinuado no ecossistema React) | — | ❌                  | **Não recomendado**                           |

**Diretriz prática:** Tailwind v4 para estilização geral, CSS Modules para estilos que ficam mais claros como código próprio, e Bootstrap apenas quando o time já tem afinidade com ele.

### 6. Exemplo integrando duas abordagens (Tailwind + CSS Module)

```jsx
// app/_components/Card.js
import styles from './Card.module.css';

export default function Card({ title, children }) {
  return (
    <article className={`${styles.card} bg-white shadow-md rounded-lg p-6`}>
      <h3 className="mb-3 text-lg font-semibold text-gray-800">{title}</h3>
      {children}
    </article>
  );
}
```

```css
/* app/_components/Card.module.css */
.card {
  transition: transform 200ms ease;
}
.card:hover {
  transform: translateY(-4px);
}
```

Tailwind cuida das utilitárias comuns e o CSS Module cobre o efeito custom de hover. É uma ótima combinação no App Router: zero JS extra, estilos previsíveis, sem configurações adicionais.

### 7. Checklist de boas práticas

- [ ] Importar Global CSS **apenas** em `app/layout.js` (ou layouts de route group).
- [ ] Usar CSS Modules sempre que um estilo for específico de um componente.
- [ ] Com Tailwind v4, evitar `tailwind.config.js` — configurar tokens via `@theme` no CSS.
- [ ] Usar classes dinâmicas com template strings **com cuidado** (o Tailwind precisa enxergar a classe literal para gerá-la).
- [ ] Usar `clsx` ou `classnames` para combinar classes condicionais (opcional).
- [ ] Isolar bibliotecas CSS que manipulam o DOM (Bootstrap JS, Materialize) em Client Components.
- [ ] Não importar CSS de outras libs dentro de Client Components (faz o Next inlinar em cada bundle); importe sempre no layout/server-side.

### Conclusão

O Next.js 16 é flexível com estilização. O caminho mais indicado para projetos novos é o próprio **template `create-next-app@16.2.4 --tailwind`** — ele já entrega Tailwind v4 + CSS Modules + Sass-ready sem esforço. Bootstrap continua útil para protótipos, mas deve ser usado com consciência no mundo de Server Components. Materialize não é mais recomendado para projetos modernos. Em todos os casos, fixe as versões nas instalações (`npm install tailwindcss@4.2.4 @tailwindcss/postcss@4.2.4`, `npm install bootstrap@5.3.8`, etc.) para manter os tutoriais estáveis ao longo do tempo.
