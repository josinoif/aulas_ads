## 1. Fundamentacao teorica (alinhada ao CTFL e TAE)

### 1.1 O que e teste de software (e o que nao e)

Teste de software e um conjunto de atividades para avaliar qualidade, reduzir risco e produzir informacao sobre o comportamento do sistema.  
Nao existe objetivo realista de "provar ausencia total de defeitos"; o objetivo e reduzir incerteza e apoiar decisao.

### 1.2 Principios de teste (resumo pratico)

Principios classicos usados em aula:

- Teste mostra presenca de defeitos, nao prova ausencia.
- Teste exaustivo e impossivel: precisamos de selecao inteligente de casos.
- Testar cedo reduz custo de correcao.
- Defeitos tendem a se concentrar em poucas areas (defect clustering).
- Testes repetidos sem renovacao perdem eficacia (pesticide paradox).
- Teste depende de contexto (sistema, risco, negocio).

### 1.3 Processo de teste (ciclo de trabalho)

Fluxo simplificado, inspirado no processo de teste do CTFL:

1. **Planejamento**: escopo, riscos, estrategia, criterios.
2. **Monitoramento e controle**: acompanhar execucao e ajustar.
3. **Analise**: identificar condicoes de teste a partir de requisitos/regras.
4. **Projeto**: desenhar casos e dados.
5. **Implementacao**: escrever scripts/specs e preparar ambiente.
6. **Execucao**: rodar testes e registrar resultado.
7. **Conclusao**: consolidar evidencias, cobertura e riscos residuais.

### 1.4 Niveis de teste

No nosso laboratorio:

- **Unidade**: valida uma parte pequena do sistema de forma isolada (normalmente um metodo de service).  
  Aqui usamos mocks para substituir dependencias externas (ex.: repository), focando apenas na regra de negocio.  
  E o tipo mais rapido de teste e ideal para feedback continuo durante codificacao.
- **Integracao**: valida a colaboracao entre componentes reais da aplicacao (controller + service + repository + banco).  
  O objetivo e verificar se as camadas "conversam" corretamente, incluindo validacao, persistencia e leitura de dados.  
  E mais lento que unidade, mas encontra erros que mocks podem esconder (mapeamento, query, configuracao de modulo).
- **E2E**: valida o comportamento da API do ponto de vista do cliente, via HTTP, do inicio ao fim do fluxo.  
  O teste envia requisicoes reais para endpoints e verifica status, payload e efeitos finais no estado.  
  E o nivel com maior confianca funcional, mas tambem o mais custoso em tempo e manutencao.

### 1.5 Tecnicas de caixa-preta usadas na aula

- **Particoes de equivalencia**: tecnica para reduzir quantidade de testes sem perder qualidade.  
  A ideia e separar o dominio de entrada em grupos (particoes) que devem ter o mesmo comportamento.  
  Em vez de testar todos os valores possiveis, testamos 1 ou 2 representantes de cada grupo.  
  Exemplo: para `name` com tamanho entre 3 e 40, criamos particoes como "valida (3..40)", "invalida (<3)" e "invalida (>40)".
- **Analise de valor limite**: tecnica que foca nos pontos onde erros aparecem com maior frequencia: fronteiras.  
  Para cada limite, testamos valor abaixo, no limite e acima do limite.  
  Exemplo: se `stock` minimo e `0`, vale testar `-1`, `0` e `1`; se maximo e `1000`, testar `999`, `1000` e `1001`.  
  Essa tecnica complementa particoes de equivalencia, porque "aperta" justamente os pontos de transicao entre valido e invalido.
- **Tabela de decisao**: tecnica para regras com varias condicoes combinadas, evitando lacunas logicas.  
  Primeiro listamos condicoes (entradas) e possiveis resultados (acoes), depois geramos linhas com combinacoes relevantes.  
  Cada linha importante vira um caso de teste.  
  Exemplo na reserva de estoque: condicoes `quantity <= 0`, `quantity > stock`, `quantity == stock`; acoes esperadas "rejeitar" ou "permitir e zerar".  
  Isso ajuda a garantir cobertura de regra de negocio complexa sem depender de tentativa e erro.

### 1.6 Risco, cobertura e automacao (visao TAE)

Do ponto de vista de engenharia de automacao:

- Nao automatizar tudo indiscriminadamente; automatizar o que tem alto retorno.
- Priorizar cenarios com alto risco de negocio e alta frequencia de regressao.
- Medir cobertura, tempo de execucao, flakiness e custo de manutencao.
- Manter arquitetura de testes legivel e estavel (facil de evoluir).

## 3. Ferramentas centrais: Jest e Supertest

No ecossistema NestJS, duas bibliotecas aparecem o tempo todo em testes:

- **Jest**: framework de testes (runner, assercoes, mocks, cobertura).
- **Supertest**: cliente HTTP de teste para validar endpoints.

### 3.1 O que e o Jest e para que serve

Jest e a ferramenta que:

- encontra e executa arquivos de teste (`*.spec.ts`, `*.e2e-spec.ts`);
- oferece API de assercoes (`expect(...).toBe(...)`, `toEqual`, `toThrow`);
- permite mocks e spies (`jest.fn`, `jest.spyOn`);
- gera cobertura de codigo (`npm run test:cov`).

No NestJS, ele ja vem como padrao no projeto criado com CLI, o que facilita adocao.

### 3.2 O que e o Supertest e para que serve

Supertest permite simular requisicoes HTTP sem precisar abrir cliente externo (Postman/Insomnia) durante o teste automatizado.

Com ele, voce consegue:

- enviar `GET`, `POST`, `PATCH`, `DELETE`;
- verificar status code e headers;
- validar corpo da resposta;
- testar fluxos de API de forma repetivel em CI.

Exemplo simples:

```typescript
import { INestApplication } from '@nestjs/common';
import * as request from 'supertest';

declare const app: INestApplication;

await request(app.getHttpServer())
  .post('/products')
  .send({ name: 'Mouse', sku: 'MOU-1', price: 100, stock: 2 })
  .expect(201);
```

### 3.3 Como Jest e Supertest se complementam

- Jest organiza e orquestra os testes.
- Supertest executa chamadas HTTP dentro desses testes.

Em outras palavras: **Jest e o "motor de testes"** e **Supertest e a "ferramenta de requisicao"** para testar API.

### 3.4 Facilidades que eles trazem para um projeto NestJS

Principais ganhos praticos:

- Feedback rapido durante desenvolvimento (`test:watch`).
- Regressao automatica a cada alteracao.
- Testes reproduziveis no CI.
- Menor dependencia de teste manual repetitivo.
- Diagnostico mais claro de erro (qual caso falhou e por que).
- Evolucao mais segura de regras de negocio.

## 4. Entendendo o codigo antes de testar

### 4.1 `AppModule`: infraestrutura da aplicacao

No `src/app.module.ts`, o `TypeOrmModule.forRootAsync(...)`:

- Le variaveis de ambiente (`DB_HOST`, `DB_PORT`, etc.).
- Configura conexao com MySQL.
- Habilita `autoLoadEntities`, para registrar entidades automaticamente.
- Usa `synchronize: true` para ambiente didatico (nao recomendado em producao).

**Por que isso importa para testes?**  
Porque testes de integracao/e2e dependem da conexao real e da estrutura de tabela.

### 4.2 `Product` entity: contrato com o banco

No `src/products/product.entity.ts`:

- `@Entity('products')` define a tabela.
- `@PrimaryGeneratedColumn()` gera `id` automaticamente.
- `sku` possui `unique: true` (restricao de unicidade no banco).
- `price` usa decimal para valores monetarios.

**Por que isso importa para testes?**  
Porque parte das regras pode ser validada no service e parte no proprio banco.

### 4.3 `ProductsService`: onde as regras vivem

No `src/products/products.service.ts`, os metodos `create`, `updateStock` e `reserve` concentram regras de negocio.  
Algumas delas estao propositalmente erradas para o aluno detectar com testes.

**Por que isso importa para testes?**  
Porque o principal objetivo e transformar regra de negocio em casos testaveis.

### 4.4 `ProductsController`: contrato HTTP

No `src/products/products.controller.ts`:

- Endpoints recebem payload, parametros e repassam ao service.
- E o ponto de entrada para testes de integracao/e2e.

**Por que isso importa para testes?**  
Porque o cliente usa HTTP; logo, validamos status code, resposta e efeitos no estado.

1. SKU deve ser unico.

Essa "falha inicial" e parte do aprendizado: primeiro demonstrar o defeito, depois corrigir.

## 7. Passo a passo pratico de escrita de testes

### 7.1 Etapa A - Testes de unidade (rapidos, foco em regra)

### Objetivo

Detectar desvios logicos no service sem dependencia de HTTP.

### Passo A1 - Criar arquivo de teste

Crie `src/products/products.service.spec.ts`.

### Passo A2 - Montar estrutura AAA

Use o padrao:

- **Arrange**: prepara estado.
- **Act**: executa acao.
- **Assert**: verifica resultado.

### Passo A3 - Escrever primeiro caso para regra de preco

```typescript
import { Test } from '@nestjs/testing';
import { getRepositoryToken } from '@nestjs/typeorm';
import { Repository } from 'typeorm';
import { ProductsService } from './products.service';
import { Product } from './product.entity';

describe('ProductsService', () => {
  let service: ProductsService;
  let repo: jest.Mocked<Repository<Product>>;

  beforeEach(async () => {
    const moduleRef = await Test.createTestingModule({
      providers: [
        ProductsService,
        {
          provide: getRepositoryToken(Product),
          useValue: {
            create: jest.fn(),
            save: jest.fn(),
          },
        },
      ],
    }).compile();

    service = moduleRef.get(ProductsService);
    repo = moduleRef.get(getRepositoryToken(Product));
  });

  it('deve manter o preco informado no cadastro', async () => {
    repo.create.mockImplementation((dto) => dto as Product);
    repo.save.mockImplementation(async (entity) => ({ id: 1, ...entity } as Product));

    const created = await service.create({
      name: 'Notebook',
      sku: 'NOTE-001',
      price: 1500,
      stock: 5,
    });

    expect(created.price).toBe(1500);
  });
});
```

#### Leitura guiada do codigo

- `Test.createTestingModule(...)`: cria um modulo isolado de teste, sem subir servidor HTTP.
- `provide: getRepositoryToken(Product)`: substitui o repositorio real por um mock controlado no teste.
- `useValue: { ... }`: diz ao Nest DI para injetar esse objeto fake no lugar da dependencia real.
- `jest.fn()`: cria funcoes mockaveis/observaveis; com elas podemos:
  - definir retornos (`mockReturnValue`, `mockResolvedValue`);
  - simular erros (`mockRejectedValue`);
  - verificar chamadas (`toHaveBeenCalledWith`, quantidade de chamadas, ordem).
- `repo.create.mockImplementation(...)`: define como a dependencia deve se comportar quando chamada.
- `repo.save.mockImplementation(...)`: simula persistencia retornando objeto com `id`.
- `service.create(...)`: executa a regra de negocio que queremos validar.
- `expect(created.price).toBe(1500)`: define o oracle (resultado esperado segundo a regra).

Em resumo:

- `useValue` troca a implementacao real por uma dublagem no contexto do teste.
- `jest.fn()` transforma metodos dessa dublagem em "espioes" e "simuladores", dando controle total sobre o comportamento da dependencia.

Como interpretar o resultado:

- **Verde**: a implementacao manteve o preco como informado.
- **Vermelho**: existe divergencia entre regra de negocio e codigo atual.

### Passo A4 - Cobrir estoque negativo

Escreva caso que aplique delta negativo e verifique rejeicao.  
Esse teste ajuda a capturar erros de fronteira e validacao de estado.

### 7.2 Etapa B - Testes de integracao (HTTP + service + MySQL)

Validar colaboracao entre camadas reais, inclusive persistencia.

### Passo B1 - Criar arquivo

### Passo B2 - Subir app de teste

```typescript
import { INestApplication, ValidationPipe } from '@nestjs/common';
import { Test } from '@nestjs/testing';
```

#### Leitura guiada do setup

- `await app.init()`: conclui bootstrap e deixa rotas prontas para o Supertest.

### Passo B3 - Criar caso de persistencia real

```typescript
import * as request from 'supertest';
import { INestApplication } from '@nestjs/common';

declare const app: INestApplication;

it('POST seguido de GET deve refletir item persistido', async () => {
  await request(app.getHttpServer())
    .post('/products')
    .send({ name: 'Mouse', sku: 'MOU-100', price: 120, stock: 10 })
    .expect(201);

  const list = await request(app.getHttpServer()).get('/products').expect(200);
  expect(list.body.some((p: any) => p.sku === 'MOU-100')).toBe(true);
});
```

#### Leitura guiada do caso

- Primeiro `POST /products`: cria dado real no banco de teste.
- Depois `GET /products`: valida se o dado persistiu e pode ser recuperado.
- `some((p) => p.sku === 'MOU-100')`: checa comportamento de negocio observado pelo cliente.

Se falhar nesse ponto, o problema costuma estar na integracao entre camadas (controller/service/repository/banco), e nao apenas em logica isolada.

### 7.3 Etapa C - E2E orientado a fluxo de negocio

### Objetivo

Validar jornada completa, incluindo regra de reserva em limite.

### Passo C1 - Criar produto

Fazer `POST /products` e capturar `id`.

### Passo C2 - Reservar exatamente o estoque

Fazer `PATCH /products/:id/reserve` com quantidade igual ao estoque.

### Passo C3 - Verificar resultado

Esperar `200` e `stock` final igual a `0`.

Esse caso e didaticamente importante porque exercita valor limite e regra de negocio critica.

Leitura pedagogica do fluxo:

- C1 garante pre-condicao (produto existente).
- C2 executa acao principal (reserva no limite).
- C3 valida pos-condicao (estoque esperado).

Esse formato ajuda o aluno a pensar em **estado antes -> acao -> estado depois**, que e o nucleo de testes de regra de negocio.

Exemplo completo da etapa C:

```typescript
import { INestApplication, ValidationPipe } from '@nestjs/common';
import { Test } from '@nestjs/testing';
import * as request from 'supertest';
import { AppModule } from '../src/app.module';

describe('Products e2e - reserva no limite', () => {
  let app: INestApplication;

  beforeAll(async () => {
    const moduleRef = await Test.createTestingModule({
      imports: [AppModule],
    }).compile();

    app = moduleRef.createNestApplication();
    app.useGlobalPipes(new ValidationPipe({ whitelist: true, transform: true }));
    await app.init();
  });

  afterAll(async () => {
    await app.close();
  });

  it('deve reservar exatamente o estoque e zerar saldo', async () => {
    const created = await request(app.getHttpServer())
      .post('/products')
      .send({ name: 'Teclado', sku: `TEC-${Date.now()}`, price: 200, stock: 4 })
      .expect(201);

    const id = created.body.id;

    await request(app.getHttpServer())
      .patch(`/products/${id}/reserve`)
      .send({ quantity: 4 })
      .expect(200);

    const fetched = await request(app.getHttpServer())
      .get(`/products/${id}`)
      .expect(200);

    expect(fetched.body.stock).toBe(0);
  });
});
```

## 8. Exemplos essenciais (o que voce pediu)

Esta secao traz exemplos prontos para copiar e adaptar no projeto-base.

### 8.1 Como testar um endpoint

Exemplo: testar `GET /products/:id` retornando `200`.

```typescript
import { INestApplication } from '@nestjs/common';
import * as request from 'supertest';

declare const app: INestApplication;

it('GET /products/:id deve retornar 200 e o produto', async () => {
  const created = await request(app.getHttpServer())
    .post('/products')
    .send({ name: 'Mouse Gamer', sku: 'MOU-777', price: 350, stock: 7 })
    .expect(201);

  const id = created.body.id;

  const response = await request(app.getHttpServer())
    .get(`/products/${id}`)
    .expect(200);

  expect(response.body.id).toBe(id);
  expect(response.body.sku).toBe('MOU-777');
});
```

Leitura guiada:

- O teste comeca criando um recurso para evitar dependencia de dados preexistentes.
- Em seguida, consulta esse recurso por `id`, exercitando rota parametrizada.
- Os asserts validam dois niveis: protocolo (`200`) e conteudo (`id`, `sku`).

Esse padrao evita testes frageis e deixa claro o comportamento esperado da API.

### 8.2 Como testar se validacao levanta erro esperado

Exemplo: DTO exige `name` com tamanho minimo 3.  
Teste manda `name` invalido e espera `400`.

```typescript
import { INestApplication } from '@nestjs/common';
import * as request from 'supertest';

declare const app: INestApplication;

it('POST /products deve retornar 400 para name menor que 3', async () => {
  const response = await request(app.getHttpServer())
    .post('/products')
    .send({ name: 'AB', sku: 'SKU-VAL-1', price: 100, stock: 5 })
    .expect(400);

  expect(response.body.message).toEqual(
    expect.arrayContaining([expect.stringContaining('name')]),
  );
});
```

Leitura guiada:

- O payload usa `name: 'AB'`, que viola a regra minima de tamanho.
- `expect(400)` valida que a API rejeitou corretamente a entrada invalida.
- `expect.arrayContaining(...)` confirma que o corpo de erro contem evidencia da falha de validacao.

Resultado pratico: o aluno aprende a testar nao apenas "deu erro", mas **se o erro certo foi emitido**.

### 8.3 Como fazer mock de dependencias

Exemplo unitario de `ProductsService` com mock do repositorio TypeORM.

```typescript
import { Test } from '@nestjs/testing';
import { getRepositoryToken } from '@nestjs/typeorm';
import { Repository } from 'typeorm';
import { ProductsService } from './products.service';
import { Product } from './product.entity';

describe('ProductsService (mock de dependencias)', () => {
  let service: ProductsService;
  let repo: jest.Mocked<Repository<Product>>;

  beforeEach(async () => {
    const moduleRef = await Test.createTestingModule({
      providers: [
        ProductsService,
        {
          provide: getRepositoryToken(Product),
          useValue: {
            create: jest.fn(),
            save: jest.fn(),
            findOne: jest.fn(),
            find: jest.fn(),
            clear: jest.fn(),
          },
        },
      ],
    }).compile();

    service = moduleRef.get(ProductsService);
    repo = moduleRef.get(getRepositoryToken(Product));
  });

  it('deve chamar repository.save ao criar produto', async () => {
    const dto = { name: 'Headset', sku: 'HEA-001', price: 200, stock: 8 };
    const entity = { id: 1, ...dto } as Product;

    repo.create.mockReturnValue(entity);
    repo.save.mockResolvedValue(entity);

    const result = await service.create(dto);

    expect(repo.create).toHaveBeenCalledWith(dto);
    expect(repo.save).toHaveBeenCalledWith(entity);
    expect(result.id).toBe(1);
  });
});
```

Leitura guiada:

- `useValue` injeta um repositorio fake no lugar do repositorio real do TypeORM.
- `repo.create` e `repo.save` sao controlados pelo teste, permitindo simular cenarios exatos.
- `toHaveBeenCalledWith` valida contrato de colaboracao entre service e dependencia.

Esse tipo de teste e ideal para diagnosticar regra de negocio com feedback rapido.

### 9.1 Particoes de equivalencia (exemplo: `name`)

### 9.2 Valor limite (boundary)

Casos minimos recomendados:

- 2, 3, 4
- 39, 40, 41

```typescript
import { INestApplication } from '@nestjs/common';
import * as request from 'supertest';

declare const app: INestApplication;

it.each([
  ['AB', 400],
  ['ABC', 201],
  ['A'.repeat(40), 201],
  ['A'.repeat(41), 400],

  await request(app.getHttpServer())
    .post('/products')
    .send({ name, sku: `SKU-${Date.now()}`, price: 100, stock: 1 })
    .expect(status);
});
```

## 10. Mocks x banco real: criterio de escolha

### Usar mock quando

- Quer validar regra isolada rapidamente.
- Quer simular excecao dificil de reproduzir no ambiente.
- Quer feedback curto no ciclo Red-Green.

### Usar banco real quando

- Quer validar mapeamento e constraints TypeORM/MySQL.
- Quer validar consistencia de estado em multiplas operacoes.
- Quer reduzir risco de surpresa em producao.

Melhor estrategia de curso: combinar ambos, nao escolher apenas um.

## 11. Analise de resultados (mais que "passou/falhou")

Depois da execucao, discutir:

- Quais testes falharam por regra de negocio?
- Quais falharam por problema de ambiente/dados?
- Ha sinais de teste instavel (flaky)?
- Cobertura baixa coincide com modulos mais criticos?

### Cobertura

`npm run test:cov` gera metricas:

- `statements`
- `branches`
- `functions`
- `lines`

Meta inicial razoavel:

```text
Statements: 70%
Branches:   60%
Functions:  70%
Lines:      70%
```

Mas o criterio principal e relevancia dos cenarios, nao numero isolado.

### 11.1 Como diferenciar erro de ambiente x erro de regra

- **Erro de ambiente**: conexao com MySQL falhou, dependencia nao instalada, porta ocupada, app nao inicializa.
- **Erro de regra**: app sobe, teste executa fluxo, mas resultado funcional diverge do esperado.

Pergunta rapida de diagnostico:

- Falhou no bootstrap/setup? Provavel ambiente.
- Falhou no `expect` de comportamento? Provavel regra de negocio.

### 11.2 Guia rapido de diagnostico

- **Sintoma**: `Cannot find module ...`
  - Causa provavel: `npm install` nao executado no diretorio correto.
  - Acao: rodar `npm install` em `backend/nest/projeto-base-testes` e repetir `npm run build`.
- **Sintoma**: `ECONNREFUSED 127.0.0.1:3306`
  - Causa provavel: MySQL parado ou nao pronto.
  - Acao: `docker compose up -d` + conferir `docker compose ps` e logs.
- **Sintoma**: `ER_ACCESS_DENIED_ERROR` no MySQL
  - Causa provavel: credenciais do `.env` divergentes.
  - Acao: comparar `.env` com `docker-compose.yml` e reiniciar app.
- **Sintoma**: teste de regra falha mas setup passa
  - Causa provavel: bug de regra de negocio (esperado no laboratorio).
  - Acao: registrar defeito e propor correcao guiada por teste.

### 11.3 Matriz rapida: ambiente vs regra

- **Ambiente (infra/setup)**:
  - indicadores: falha no `npm install`, `build`, bootstrap da app, conexao MySQL;
  - acao: corrigir setup antes de continuar.
- **Regra de negocio (didatico)**:
  - indicadores: teste roda ate o fim, mas falha em `expect` de comportamento;
  - acao: registrar defeito funcional e evoluir via TDD.
